# Prototype README, Patient Side Sponsorship Concept Test

Next.js (App Router) + TypeScript concept test prototype implementing
`deliverables/experiment_spec.md`: a four arm, single exposure, between subjects
concept test (Control / Variant A / Variant B / Variant C), a trust survey with a
disclosure comprehension check, and an admin results dashboard.

This app owns everything under `app/`. It does not modify any files outside this
directory. `experiment_spec.md` is the integration contract for events/properties,
survey items, randomization, and privacy, this prototype implements it, it does not
redefine it. Every place the spec left a gap or an unconfirmed number is flagged below
and in code comments, never resolved silently.

## Running locally

```bash
cd app
npm install
npm run dev
```

Open `http://localhost:3000`. The flow is: `/` (Intro) → `/concept` (assigned variant)
→ `/action` (behavioral choice) → `/survey` (trust survey) → `/thank you`. The results
dashboard is at `/admin` (no auth, see Known Limitations).

To reset your own sticky assignment during testing, clear this site's localStorage
(devtools → Application → Local Storage) or use a private/incognito window, the app
does not currently provide an in UI "reset" control, since a real participant should
only ever see one arm.

Optional query params on the Intro page for testing `participant_type`:
`http://localhost:3000/?participant_type=panel` or `?participant_type=landing_page`
(defaults to `landing_page` if omitted).

### Build

```bash
npm run build
npm run start
```

`npm run build` passes cleanly (Next.js typecheck + all 6 routes prerendered:
`/`, `/concept`, `/action`, `/survey`, `/thank you`, `/admin`). `npx eslint .` is also
clean.

Note for anyone building inside a FUSE mounted / network backed working directory:
Next.js's Turbopack build can raise a `Bus error` under `mmap` on some FUSE mounts. If
that happens, `rsync` the project (excluding `node_modules` and `.next`) to local disk,
run `npm install && npm run build` there, then discard the build output, nothing under
`.next/` needs to be committed or persisted (it's gitignored).

### Regenerating synthetic data

```bash
npm run gen:synthetic
```

Regenerates `src/data/synthetic participants.json` deterministically (seeded PRNG) from
the parameters in `scripts/generate synthetic data.mjs`. ≈90 fabricated records per arm
(360 total), matching the spec's discovery scale sample size (§7).

## Plugging in a real PostHog project

Copy `.env.example` to `.env.local` and set:

```
NEXT_PUBLIC_POSTHOG_KEY=phc_your_project_key
NEXT_PUBLIC_POSTHOG_HOST=https://us.i.posthog.com # or your self hosted host
```

With no key set, the app runs fully offline: PostHog is never loaded, and every event
is instead written to `localStorage` (see `src/lib/storage.ts`), so the full flow and
the admin dashboard both work without any external service. This is intentional per the
brief ("if absent, a graceful no op wrapper... so the full flow and dashboard work
offline").

**Design choice, flagged:** even when a real key *is* set, the prototype still mirrors
every event to `localStorage` in addition to forwarding it to PostHog (`src/lib/
analytics.ts`, `capture()`). This is so the admin dashboard's "real captured data"
panel works in either configuration during a demo/review, without needing to wire the
dashboard up to query the PostHog API. It is a prototype convenience, not something
implied by the spec, a production build would instead have the admin dashboard read
aggregates from PostHog (or a warehouse synced from it) rather than from the browser's
own localStorage.

Before a real field test: session replay and autocapture are disabled at `init()`
(`disable_session_recording: true`, `autocapture: false`, `capture_pageview: false`),
matching spec §4/§8's requirement that replay and network recording be off in the
PostHog project config. Confirm the same is true in the PostHog project settings
themselves, since project level config can override client side init flags.

## What is synthetic vs. real

* `src/data/synthetic participants.json`, 360 fully fabricated records (90/arm),
 generated by `scripts/generate synthetic data.mjs` with a seeded PRNG. Not derived
 from any real Legion, patient, or panel data. Parameters were chosen to be
 *illustrative* of the pattern the spec's own hypotheses anticipate (Control best,
 Variant C most concerning), they are not a forecast.
* Anything a real participant does in the running app (`/` → `/thank you`) is captured
 as a real local record via `src/lib/storage.ts` and merged into the same dashboard,
 clearly separated by an "Include synthetic data" toggle on `/admin`.
* The `/admin` dashboard always shows a "SYNTHETIC DATA" banner regardless of the toggle
 state, since the underlying dataset defaults to including the fabricated records.

## Known limitations (prototype specific, in addition to spec §9)

1. **No backend, no real PostHog feature flag.** The spec calls for a PostHog
 multivariate feature flag as the randomization mechanism. This prototype implements
 equivalent behavior (sticky, equal allocation, client keyed assignment) entirely in
 `src/lib/participant.ts` using `localStorage` + client side randomness, since there's
 no backend or feature flag service wired up here (per the "no over building"
 constraint). A production deployment should move allocation to an actual PostHog
 feature flag so allocation ratios and rollout are controlled centrally, not by
 redeploying this code.
2. **No duplicate participant detection.** Spec §3 requires excluding participants who
 re enter with the same `distinct_id`/panel ID. This prototype's assignment is sticky
 *within a browser* (same localStorage), but a participant who clears storage or
 switches browsers/devices is silently re randomized and re counted. A real field
 test needs this handled by the panel vendor and/or a server side distinct_id check.
3. **`/admin` has no authentication.** Per the "no auth" constraint for this prototype,
 the results dashboard is reachable by anyone who knows the URL. Fine for a local/demo
 build; not acceptable for a fielded test with real (even non PHI) aggregate data.
4. **No automated qualitative coding.** Free text responses are captured and capped at
 280 characters (spec §5) but the two coder, kappa scored severity coding process in
 spec §6 is a human process this prototype doesn't attempt to automate, it only
 stores the raw text locally for that process to consume later.
5. **CSV export is aggregate only**, per spec §8's "aggregate reporting only, no
 participant level drill down", there is intentionally no participant level export.

## Handoff

### Key Findings (what building it surfaced)

* **The discovery scale sample size genuinely produces "inconclusive" results most of
 the time**, exactly as spec §7 predicts analytically. Running the dashboard against
 the seeded synthetic dataset (≈90/arm) produces a mix of `inconclusive`/`pass`/`fail`
 labels across the 9 primary vs. Control comparisons (3 metrics × 3 arms), Variant C
 is the only arm that produces `fail` labels in this synthetic run, and even some of
 Variant C's comparisons land as `inconclusive` despite a clearly simulated large
 effect, purely from small n noise. This is a useful, concrete confirmation that the
 spec's own worked power calculations (§7) aren't just theoretical, the Newcombe/
 normal approx CIs implemented here behave the way the spec says they will.
* **The nine property PostHog allow list is a real, load bearing constraint**, not just
 a style rule. Implementing it forced a genuine architectural split: Q2 (self reported
 comprehension), Q6 (paid option preference), the factual comprehension check, and the
 free text box all had to live in a separate local only data path, because none of
 them fit the nine approved properties. This directly validates the gap the spec
 itself flags in §5 and Recommendations #3, it is not something this build invented.
* **Randomization without a backend is workable but fragile at the edges.** A pure
 client side, localStorage keyed sticky assignment satisfies "equal allocation" and
 "sticky within a test window" for a single browser participant, but has no answer for
 cross device re entry or storage clearing, exactly the duplicate participant problem
 spec §3 calls out as a hard exclusion criterion, not a nice to have.

### Decision Implications

This prototype is an instrument, not a result. It does not itself tell Legion whether
to proceed with sponsorship, it's the tool that would produce the numbers
`decision_framework.md`'s Patient Gate depends on, once run against real panel/
landing page participants instead of the synthetic seed. Given §7's power analysis
(confirmed empirically above), the discovery scale run this prototype is built for
should be reported to stakeholders as a **large problem screen**, not a statistically
validated non inferiority result, a "pass" label at n≈90/arm is directionally
reassuring, not confirmatory, and should not be used alone to greenlight a live pilot
without the ≈400/arm inference scale replication the spec describes in §7.

### Remaining Unknowns / Spec Ambiguities Flagged

These are gaps in the lead approved spec that this build implemented a working
(unconfirmed) answer for, rather than resolving silently, all also called out inline
in code comments and in `analytics/event_schema.md`:

1. **H3c (independence) has no lead approved non inferiority margin.** This prototype
 reuses the spec's own proposed 0.25 point placeholder (same as trust). See
 `src/lib/metrics.ts`, `INDEPENDENCE_MARGIN_PLACEHOLDER`.
2. **Guardrail thresholds for privacy concern, opt out rate, and "strong majority"
 preferring a paid option have no lead approved numbers.** This prototype implements
 working operational definitions (privacy concern: >0.5 point delta vs. Control;
 opt out: >10pp delta vs. Control; paid preference: >50% per the spec's own
 suggested starting point) purely so the dashboard is executable, all three are
 visibly labeled "unconfirmed"/"prototype only" in the dashboard UI itself.
3. **Control arm comprehension check wording.** Spec §5 recommends (but does not
 confirm) substituting a parallel "no sponsorship" comprehension question for
 Control, instead of skipping the check. This prototype implements that
 recommendation (`src/lib/survey.ts`, `getComprehensionCheck`), flagged for lead
 confirmation before a real field test, since it affects cross arm comparability of
 the one guardrail with a hard numeric threshold (≥80%).
4. **`participant_type` (panel vs. landing_page) is set via a URL query param**, since
 this prototype has no real recruitment channel integration (panel vendor or a
 separate landing page). This stands in for the spec's channel stratified allocation
 (§3) but does not implement actual stratified randomization within channel, with
 only a prototype's worth of traffic this wasn't judged worth building; a real field
 test should stratify allocation by channel as §3 specifies.
5. **Dual write analytics (local mirror + PostHog) even when a real key is configured**
 is a prototype convenience (see "Plugging in a real PostHog project" above), not a
 spec requirement, flagged so it isn't mistaken for the intended production
 architecture.

### Recommended Next Action

Get lead sign off on the four flagged numeric thresholds above (independence margin,
privacy concern delta, opt out delta, paid preference majority) and the Control
comprehension check wording, these are cheap to confirm now and expensive to discover
are wrong after a real field run. In parallel, wire a real PostHog project
(`NEXT_PUBLIC_POSTHOG_KEY`), replace the synthetic dataset toggle with a live data only
view for the actual discovery scale test, and route real participants through a panel
vendor or landing page per spec §3, at that point this prototype is ready to run the
discovery scale (75, 100/arm) field test the spec describes in §7.

## Deploying to Vercel

1. Push this `app/` directory to a Git repo (or a subdirectory of a monorepo, set
 Vercel's "Root Directory" to `app/` in that case).
2. In Vercel: New Project → import the repo → framework preset auto detects Next.js.
3. Add environment variables (Project Settings → Environment Variables):
 * `NEXT_PUBLIC_POSTHOG_KEY` (optional, omit to run in offline/localStorage mode)
 * `NEXT_PUBLIC_POSTHOG_HOST` (optional, defaults to `https://us.i.posthog.com`)
4. Deploy. No database, no server side secrets, and no additional infrastructure are
 required, this is a static friendly Next.js app with client only state.
